# Sentiment pipeline → database migration

Migrates the project's CSV + JSON files into a relational database.
Built on SQLite (zero-setup, the 3.6 MB of data fits trivially); the schema is
written to be **PostgreSQL-portable** so you can promote it to a server later
without redesign.

## Files

| File | Purpose |
|------|---------|
| `2026-06-22-schema.sql` | Table definitions: PKs, FKs, CHECK constraints, indexes |
| `2026-06-22-migrate.py` | Idempotent loader: reads the source files, normalizes types, upserts |
| `sentiment.db` | The built SQLite database (ready to open) |

## Schema

| Table | Source file(s) | Key | Notes |
|-------|----------------|-----|-------|
| `articles` | `articles_*.json` (7 files) | `id` | Deduplicated — 3,800 raw records → 3,769 unique. `scope`→`ticker`, `date`→`published_date` |
| `article_scores` | `scored_history.csv` | `(id, snap_date)` | FK `id` → `articles.id`. Probabilities CHECK-constrained to [0,1], `sent` to [-1,1] |
| `snapshots` | `snapshots.csv` | `(date, ticker)` | Live daily signals. Empty confidence bounds stored as NULL (not 0) |
| `universe_calls` | `universe_calls.csv` | `(date, ticker)` | Backtest universe + realized returns/residuals (superset of snapshots) |
| `backtest_pnl` | `combined_book_pnl.csv` | `window` | Per-window book returns |

`snapshots` and `universe_calls` are kept separate by design: they share columns
but have zero key overlap and different semantics (live signal log vs. labeled
backtest set).

## Running it

```bash
cd db
python3 2026-06-22-migrate.py                 # builds ./sentiment.db
python3 2026-06-22-migrate.py --db /path/x.db # custom location
```

The loader is **idempotent**: every insert is `ON CONFLICT … DO UPDATE` on the
natural key, so re-running your upstream pipeline and re-loading never
duplicates rows — it refreshes them. Each table load is wrapped in a
transaction; a failure rolls that table back rather than leaving a partial load.
Foreign keys are enforced (`PRAGMA foreign_keys = ON`).

## Querying

```bash
sqlite3 sentiment.db
```
```sql
-- a ticker's scored articles with the article text
SELECT a.published_date, a.source, s.sent, a.text
FROM article_scores s JOIN articles a ON s.id = a.id
WHERE a.ticker = 'TCS.NS'
ORDER BY a.published_date DESC;

-- divergent signals
SELECT date, ticker, agg_sent, agg_news, label
FROM snapshots WHERE label LIKE '%DIVERGENT%';
```

## Verification performed after load

- Row counts match the source files exactly (articles 3,769; scores 130; snapshots 8; universe_calls 528; pnl 11).
- `PRAGMA foreign_key_check` → no violations; zero orphaned scores.
- `PRAGMA integrity_check` → ok.
- No out-of-range probabilities/sentiment; NULLs preserved (not coerced to 0).

## Pointing the pipeline at the DB

Your scripts currently read/write CSV/JSON. To move them onto the DB, replace
file reads with queries and file writes with the same upsert pattern the loader
uses. Use parameterized queries (`?` placeholders) — never f-string SQL.

```python
import sqlite3
con = sqlite3.connect("db/sentiment.db")
con.execute("PRAGMA foreign_keys = ON;")
df_scores = con.execute(
    "SELECT * FROM article_scores WHERE scope = ?", ("TCS.NS",)
).fetchall()
```

## Promoting to PostgreSQL

1. Run `2026-06-22-schema.sql` against your Postgres database unchanged.
2. In the loader, swap `sqlite3` for `psycopg`, change `?` placeholders to `%s`.
   The `ON CONFLICT … DO UPDATE … excluded.*` syntax is valid in both engines.
3. Drop the `PRAGMA foreign_keys` line (Postgres enforces FKs by default).
4. Bulk-load large tables with `COPY` for speed once volumes grow.

Move to Postgres when you have concurrent writers, want network access, or your
data outgrows comfortable single-file size. Until then SQLite is sufficient.
